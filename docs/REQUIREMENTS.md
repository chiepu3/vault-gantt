## 1. 背景と目的

現行 `obsidian-task-workbench` は会社PC上でM365 Copilot（チャットボット）に対話的に作らせたもので、実用に達している一方、以下の構造的問題を抱える:

- チャットボット駆動開発の痕跡（型なしTS、全再描画、マジックナンバー散在、疑似mixin構造、自前YAMLパーサー）
- 社内運用のハードコード（デイリーノート規則、日本語UI直書き、社員番号入りデプロイパス、実業務データ入りスナップショット）
- 拡張性の限界（AI Agent API Phase3以降が現構造では進めにくい）

**目的**: 自宅環境で1からリビルドし、**OSSとして公開**し、それを会社に持ち込んで使う。会社固有情報は一切含めない。

## 2. スコープ

### 2.1 継承する機能（現行の実証済み価値）

- Markdownノート = 唯一の真実源（プラグインは編集レイヤー）という思想
- 親タスク1ファイル+サブタスク内包のモデル（内部表現はネストYAML配列に刷新。§9-Q1で決定、詳細はFR-1.2/1.3）
- Ganttビュー一式: レーンパッキング、営業日ベースのドラッグ移動/リサイズ、マーカー、親期限★、一括移動、インライン編集（IME対応）、リッチポップオーバー、空セルメニュー、タグ絞込、ズーム、遅延範囲拡張
- 作業時間（計画/実績、0.5h刻み、日毎入力、サマリー行）
- 祝日3層（自動取得/手動/特別）+ 営業日計算
- Workbenchテーブルビュー、タスク検索モーダル、埋め込みブロック
- 自動優先度（期限ベース、手動オーバーライド）
- statusLabel↔completed同期規則
- Core Task API（patch検証・ホワイトリスト・revision楽観ロック・batch更新・解析キャッシュ）
- 読み取り専用Webビューワーへのスナップショット同期

### 2.2 落とす・置き換えるもの

- Meta Bind依存のノート内ダッシュボード → **廃止**。編集はプラグインUI（Workbench/Gantt/専用パネル）に一本化し、ノート本文は最終保存値の読み取り専用サマリーのみ表示する（§9-Q2）
- デイリーノート連携のハードコード → 汎用設定化（フォルダ/ファイル名パターン/挿入見出しをユーザー定義）
- `開発指示書` などCopilot統制用の仕組み一式
- 実データ入りスナップショット・社内デプロイパス
- サーバーv1（HTML内蔵版）→ v2系（ビルド成果物配信）に一本化

### 2.3 非スコープ（初期リリースでは扱わない）

- 全履歴スナップショット・分岐管理・無制限Undo（※直近N件の**軽量Undo自体はこの非スコープに含まない**＝in-scope。FR-2.5として実装する。§9追加注文）
- モバイル最適化UI（動作はすること。isDesktopOnly: false は維持）
- 複数Vault間同期・マルチユーザー編集
- Webビューワーのコメント機能・通知連携（Discord等）→ M7として将来ロードマップに記録のみ（§9-Q6）

## 3. 機能要求（FR）

### FR-1 データレイヤー

| ID | 要求 | 優先度 |
|---|---|---|
| FR-1.1 | タスクノート形式を仕様として文書化し、パーサー/シリアライザーを純粋関数+単体テストで実装する | MUST |
| FR-1.2 | frontmatterの読みは`app.metadataCache.getFileCache(file).frontmatter`、書きは`app.fileManager.processFrontMatter()`のみで行う。**自前パーサー・外部YAMLライブラリは不要かつ禁止**（Obsidianが既にフルYAMLパース済みのオブジェクトを提供するため） | MUST |
| FR-1.3 | サブタスクは`subtasks: [{key, title, statusLabel, ...}]`の**ネストYAML配列**として単一の親ファイル内frontmatterに格納する（現行`subtask__key__field`フラットキーからの刷新）。Properties表示は`cssclass: vault-gantt-task`（後継クラス名）スコープの`.metadata-container{display:none}`で隠し、他ノート種別には影響させない（現行実装 `styles.css:125` で実証済みの手法を継承）。サブタスク別ファイル化は不採用（§9-Q1: 非表示手法が既にnote種別スコープで機能しており、分割の実利がないため） | MUST |
| FR-1.4 | 現行形式（`subtask__key__field` フラットキー）から新形式（FR-1.3）への**一方向移行コマンド**を提供する | MUST |
| FR-1.5 | ステータスは5値+日本語ラベルを既定としつつ、**ユーザー定義可能**にする | SHOULD |
| FR-1.6 | 自動優先度の閾値（5/4/3/2/1の日数境界）を設定可能にする | COULD |

### FR-2 Core Task API

| ID | 要求 | 優先度 |
|---|---|---|
| FR-2.1 | patch検証（ホワイトリスト/型/日付/整合性規則）・revision楽観ロック・batch更新を現行同等以上で再実装 | MUST |
| FR-2.2 | すべてのUI操作はCore API経由でのみ書き込む（直接vault.modify禁止の層構造） | MUST |
| FR-2.3 | Core APIの全メソッドを**JSON in/out・Obsidian型非依存**のツールコール形状で設計する。Phase1の時点で「AI Agentが後で直接呼べる形」を前提にし、Phase3のTool Adapter/dry-run/差分プレビュー/MCP連携が薄いラッパーの追加だけで済むようにする | **MUST**（初版のSHOULDから格上げ。将来のAI機能導入を前提とするため） |
| FR-2.4 | 変更イベントをUIに通知するサブスクリプション機構（全再描画からの脱却の基盤） | MUST |
| FR-2.5 | **軽量Undo**: 直近N件（既定20件、設定可）のCore API操作を`{適用前revision, 適用patch}`のセッション内スタックで保持し、Ctrl+Zで逆patchを当てて取り消す。永続化なし（アプリ再起動でクリア）・分岐なし・Redoは単純に取り消しスタックからの再適用のみの線形モデル。REVISION_CONFLICT時は取り消し自体を失敗させUIに通知する | MUST |

## 6. アーキテクチャ方針（提案）

```
src/
  domain/        # タスクモデル・パース/シリアライズ（ネストYAML、metadataCache直読み）・営業日計算・優先度・レーンパッキング・軽量Undoスタック（純TS、テスト対象、フレームワーク非依存）
  application/   # Core Task API（検証・revision・batch・変更通知・Undo）。全メソッドJSON in/out、AI Agent Tool Adapterの下地
  infra/         # VaultAdapter（processFrontMatterでの読み書き）、祝日フェッチ、同期クライアント、設定永続化
  ui/
    workbench/   # テーブルビュー（Svelte）
    gantt/       # Ganttビュー（描画エンジン+インタラクション、差分更新。vanilla TS——viewerと共有するためフレームワーク非依存を維持）
    shared/      # モーダル・ポップオーバー・設定タブ（Svelte）、キーマップ定義（NFR-9）
  main.ts        # 合成ルート（DI）
viewer/          # Webビューワー（domain+ui/gantt〈vanilla TS部分〉のみ共有、infraをHTTPスナップショットに差し替え。Workbenchを含まないためSvelteランタイムは不要）
```

- UIフレームワーク: **Svelte（workbench/shared） + vanilla TS（gantt）のハイブリッド**（§9-Q3で決定）
- 描画: Gantt差分更新は素のTSで設計（現行資産の知見を活かす）
- 状態管理: 単方向データフロー（Core APIの変更通知 → ビューが購読して差分更新）
- 現行コードは**仕様書として参照**し、コードのコピーはしない（クリーンリビルド。会社成果物の混入回避の意味でも）

## 8. 受け入れ基準（全体）

1. 現行Vaultのタスクを移行コマンドで無損失変換でき、Gantt上の見た目・操作が現行と同等
2. 親50/サブ500/180日表示でドラッグ・スクロールにもたつきがない（FR-3.3）
3. `strict: true` でコンパイル警告ゼロ、domain層テストがCIで回る
4. リポジトリ全履歴に会社固有情報が存在しない（初回コミット前に検査）
5. UIは日本語のみで全機能が使える（多言語対応は行わない）

## 9. 決定事項ログ（以前: 未決事項）

| # | 論点 | 選択肢と考察 |
|---|---|---|
| Q1 | **ノート形式を維持するか刷新するか** | (a) 現行フラットキー形式を維持=移行不要・ただし負債継続 / (b) ネストYAML or サブタスク別ファイル化=きれいだが移行コマンド必須。**推奨: (b)を新形式とし、(a)からの一方向移行を提供** |

| Q6 | **Webビューワー同期の要否** | 会社で使う前提なら必要性を再確認（社内ネット事情）。初期リリースから入れるか、M6まで遅らせるか |
| Q7 | **多言語の初期範囲** | ja/enの2言語で開始か、jaのみで開始しi18n機構だけ先に入れるか |

### 決定（2026-07-19）

| 論点 | 決定内容 | 理由 |
|---|---|---|
| Q1 | **ネストYAML配列を単一の親ファイルのfrontmatterに格納**（サブタスク別ファイル化は不採用） | 現行実装が既に`cssclass: vault-gantt-task`スコープの`.metadata-container{display:none}`でタスクノートのProperties表示だけを隠すスコープ限定CSS手法で対応済み。ネスト化してもこの仕組みをそのまま継承でき、別ファイル化の実利がないため。参照: FR-1.3 |
| Q2 | **Meta Bind依存は全廃止。編集はプラグインUIに一本化** | ノート本文は最終保存値の読み取り専用サマリーのみとし、ダッシュボード生成は不要。参照: §2.2, FR-2 |
| Q3 | **Svelte（workbench/shared）+ vanilla TS（gantt）のハイブリッド** | Svelteの近零ランタイムで軽量なUIフォーム（Workbench/モーダル/設定）。ganttはフレームワーク非依存を維持し、読み取り専用Webビューワーとの差分描画エンジン共有を可能に。参照: §6アーキテクチャ図 |
| Q4 | **プラグイン名「Vault Gantt」（id: vault-gantt）に決定** | 既定で設定済み。参照: 1行目のプラグイン名欄 |
| Q5 | **ライセンスはMIT** | 会社利用の受け入れやすさ。参照: §5 |
| Q6 | **読み取り専用Webビューワー同期は継続**（常時稼働の自宅PCで運用）| 現行の上長が時々閲覧する運用を踏襲。将来的なコメント機能・通知連携（Discord等）はM7として初期スコープ外の将来ロードマップに記録。参照: M7 |
| Q7 | **多言語対応は行わない。UIは日本語のみ。i18n機構は導入しない** | 参照: NFR-5, §8 item 5 |
